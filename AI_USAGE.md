# AI 工具使用说明

## AI 工具使用场景

本项目在开发过程中使用 Claude Code（Anthropic）作为辅助编程工具，主要应用场景包括：

- **代码生成与补全**：快速生成 React 组件、Flask 路由、类型定义、API 封装等样板代码
- **代码审查与重构**：审查菜单结构、路由配置、状态管理逻辑，提出重构建议
- **Bug 诊断**：分析登录 500 错误、端口占用、JSON 编码等问题
- **测试辅助**：编写 curl 命令批量验证登录、工单、用户、日志等接口
- **文档生成**：生成 README、Docker 部署文档、项目复盘等文档
- **架构设计建议**：讨论系统设置二级菜单、操作日志表结构、Docker 多服务编排方案

## 关键 Prompt 示例

### 1. 需求理解与任务分解

> 1、首页看板似乎有问题
> 2、状态，用户，权限管理全部放到系统设置里面，二级菜单
> 3、工单需要人可以编辑，改变工单状态

AI 据此拆分为三个任务：修复首页看板、重构系统设置菜单、优化工单编辑与状态变更。

### 2. 调试与问题定位

> 1、登陆报错了：Request failed with status code 500
> 2、登录页面图片似乎有点问题，logo看不见了

AI 通过检查后端路由、静态资源目录、端口占用情况，定位到 logo 图片未放入 `frontend/public`，以及 5173 端口被旧进程占用导致请求代理异常。

### 3. 功能扩展

> 1、登录页面左上角还是只能看到一半logo……修改结束后生成一个版本
> 2、加上操作日志，记录操作
> 3、完整测试下，看看有没有功能上的漏洞
> 4、生成docker部署文档
> 5、请同时生成 README.md、AI_USAGE.md、REVIEW.md、Dockerfile 和 docker-compose.yml

AI 据此创建任务清单，依次完成前端适配、操作日志、测试修复、Docker 部署和文档生成。

## AI 生成内容的问题

1. **细节遗漏**：
   - 首次生成 `api.ts` 的 `listLogs` 函数时漏写了 `=>` 箭头，导致 TypeScript 编译报错，需人工修正。
   - 生成 `Dashboard.tsx` 修改变量名后，未同步修改引用处，导致构建报错，需人工检查修复。

2. **路径/上下文理解偏差**：
   - 多次在 `cd frontend && npm run build` 后继续使用 `cd frontend`，因 Bash 工作目录已切换导致命令失败，需要人工确认当前目录。
   - 全局错误处理中一度遗漏 `import os`，经构建/运行测试后发现并补齐。

3. **状态同步问题**：
   - `Logs.tsx` 的搜索函数中直接调用 `fetchLogs` 而依赖的 `keyword` 状态尚未更新，AI 初版存在潜在的竞态问题，人工 review 后改为通过 `useEffect` 依赖触发。

4. **过度工程风险**：
   - AI 倾向于给出更完整的解决方案，有时会引入当前阶段不必要的复杂度。例如操作日志本可以只做后端记录，但为提升可观测性同时增加了前端日志页面，这由人工确认后保留。

## 人工修改内容

1. **目录与文件管理**：
   - 手动将 `etc/png/login-bg-with-logo.png` 复制到 `frontend/public/`，确保前端能正确加载背景图。
   - 手动创建 `.dockerignore` 文件，排除 `node_modules`、`__pycache__` 等不需要打包的内容。

2. **代码质量修正**：
   - 修复 `api.ts` 箭头函数语法错误。
   - 修复 `Dashboard.tsx` 变量重命名后的引用问题。
   - 修复 `app.py` 中遗漏的 `import os`。
   - 修复 `Logs.tsx` 搜索状态同步问题。
   - 统一 `tickets.py` 中 `updated_at` 时间格式与 SQLite 默认格式一致。

3. **架构与业务决策**：
   - 决定操作日志页面仅对超级管理员可见，使用 `role:manage` 权限控制。
   - 决定编辑工单时展示全部状态，由后端校验流转规则，提升用户体验。
   - 决定 Docker 部署中使用 Nginx 作为前端静态服务器并反向代理 `/api` 到后端。
   - 增加 `DB_PATH` 环境变量支持，便于 Docker 数据持久化。

4. **测试与验证**：
   - 手动执行后端启动、前端构建、curl 接口测试、端口占用排查等验证步骤。
   - 手动终止旧 node/python 进程，释放被占用的 5173 端口。

5. **文档打磨**：
   - 根据实际改动调整 README 中的安装、运行、Docker 部署说明。
   - 补充已知问题和后续改进项，确保文档与代码现状一致。
